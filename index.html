How the Model Works

The Core Idea

Data centers do not publish server counts. But they leave public traces in utility permits, EPA generator filings, sustainability reports, and analyst coverage. This model sources power figures from those records. The user never enters MW manually. Three variables then convert sourced power into an estimated CPU count.

Formula

Sourced DC Power (MW) ÷ PUE = IT Load (MW)
IT Load x Utilization x 1,000,000 = Effective Draw (W)
Effective Draw ÷ Weighted TDP = CPU Count

Cross-Check

Where public GPU or server counts exist (CoreWeave's disclosed GPU fleet, Nebius's 60,000 GPU Mantsala count), the model cross-checks the CPU point estimate against a derived host CPU count using the 1 CPU per 2 GPU ratio for Hopper and Blackwell systems. A badge shows alignment status.

GPU vs CPU Distinction

CPU-centric providers (Google, AWS, Meta, Azure) use CPUs as the primary compute unit for general-purpose cloud workloads. GPU-centric providers (CoreWeave, Nebius) use GPUs for AI; CPUs only exist as host controllers. The model always estimates CPU sockets specifically. For GPU-centric facilities this means a much smaller number than total compute.

Forecast Methodology

Each provider's forecast uses their own CapEx from SEC filings and earnings guidance, not a shared aggregate. AWS uses 64% of Amazon's total (the DC-attributed fraction per Platformonomics). GPU share of server spend rises from 40% in 2024 to 65% by 2030 for hyperscalers, and from 80-85% to 90% for GPU-centric providers. CPU budget is 25% of the remaining non-GPU server spend. New sockets are divided by blended CPU cost per provider's ARM/x86 mix. Fleet accumulates with 20% annual retirement on a 5-year refresh cycle.
